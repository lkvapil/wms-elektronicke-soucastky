\chapter{Datový model}
\label{chap:datovy_model}

\section{Úvod -- formulace problému}

Cílem tohoto projektu je teoreticky navrhnout informační systém pro správu skladových
součástek. Systém má umožňovat evidenci součástek, jejich kategorií, dodavatelů
a skladových lokací. Dále má podporovat sledování pohybů součástek ve skladu,
jako je naskladnění, vyskladnění nebo přesun mezi jednotlivými lokacemi.

Součástí systému je také možnost přiřazovat součástky k projektům a evidovat
jejich množství potřebné pro daný projekt. Návrh systému je znázorněn pomocí
UML class diagramu, který popisuje jednotlivé třídy, jejich atributy a vztahy
mezi nimi. Diagram využívá vybrané vlastnosti UML, jako jsou zobecnění,
kardinality, agregace, kvalifikované vazby a atributy spojení, aby byl model
systému co nejpřesnější.

\section{Přehled entit}

Systém pracuje s~následujícími hlavními třídami:

\begin{description}
  \item[\texttt{Administrator}] Hlavní uživatel systému; vytváří pohyby a spravuje součástky.
  \item[\texttt{Sklad}] Fyzický sklad identifikovaný názvem;
        sdružuje skladové lokace.
  \item[\texttt{Lokace}] Konkrétní místo v~skladu (přihrádka, regál)
        identifikované označením.
  \item[\texttt{Soucastka}] Elektronická součástka evidovaná v~systému;
        obsahuje atributy \texttt{id}, \texttt{nazev} a aktuální \texttt{mnozstvi} na skladě.
  \item[\texttt{Projekt}] Projekt, ke kterému jsou součástky přiřazovány;
        identifikován pomocí \texttt{id} a \texttt{nazev}.
  \item[\texttt{Pohyb}] Záznam o~pohybu součástky (příjem, výdej, přesun);
        obsahuje \texttt{id}, \texttt{typ} pohybu, \texttt{datum} provedení a aktuální \texttt{stav}.
\end{description}

\section{Stavový diagram -- Skladový pohyb}

Třída \texttt{Pohyb} reprezentuje obecný skladový pohyb -- tedy jakoukoli změnu stavu zásob součástek
v systému. Atribut \texttt{typ} rozlišuje tři druhy pohybu: \emph{příjem} (naskladnění nových součástek),
\emph{výdej} (odebrání součástek ze skladu) a \emph{přesun} (přemístění součástek mezi lokacemi či sklady).
Každý pohyb prochází životním cyklem: po vytvoření je odeslán k~validaci (\emph{Validován}), při které
systém ověří, zda zadaná součástka existuje a je dostupná na skladě. Pokud jsou data platná, pohyb přejde
do stavu \emph{Schválen} a může být fyzicky proveden -- dojde k~aktualizaci množství na skladě.
Pohyb může být zamítnut ve dvou situacích:
\begin{itemize}
  \item při validaci (\texttt{[neplatna\_data]}) -- součástka s daným ID neexistuje v systému,
        chybí povinné údaje (typ pohybu, cílová lokace, datum) nebo zadané množství je 0 či záporné;
  \item po schválení (\texttt{[nedostatecne\_mnozstvi]}) -- požadované množství výdeje je vyšší
        než aktuální stav součástky na skladě.
\end{itemize}

\section{Scénáře use case}

\subsection*{Scénář: Vytvoření pohybu}
\begin{enumerate}
  \item Administrator zvolí \textit{Vytvořit pohyb}.
  \item Zadá součástku a množství.
  \item Systém provede kontrolu dat.
  \item Pokud \texttt{[data\_ok]} $\rightarrow$ pohyb je schválen.
  \item Systém provede pohyb a aktualizuje stav skladu.
  \item Zobrazí se potvrzení výsledku.
\end{enumerate}

\subsection*{Scénář: Přiřazení součástky k~projektu}
\begin{enumerate}
  \item Administrator vybere projekt.
  \item Zvolí součástku a zadá množství.
  \item Systém ověří dostupnost voláním \texttt{overDostupnost()}.
  \item Pokud \texttt{[dostatek]} $\rightarrow$ součástka je přiřazena.
  \item Systém aktualizuje množství na skladě.
\end{enumerate}
